## 二、架构设计原则

| 原则 | 说明 |
|------|------|
| **分层解耦** | 前后端分离，各层职责清晰，通过 API 契约通信 |
| **模块化单体** | 初期采用模块化单体架构，降低运维复杂度，预留微服务拆分能力 |
| **多端复用** | 前端以 React 为统一技术栈，最大化代码共享 |
| **开源优先** | 优先使用社区成熟、长期维护、文档完善的开源框架；业务侧只做薄封装与适配，不重复造轮子 |
| **安全纵深** | 认证、授权、传输加密、数据加密多层防护 |
| **可观测性** | 全链路日志、指标监控、告警体系 |
| **渐进演进** | 架构支持从单体到分布式的平滑演进 |

**能力选型优先级：**
1. 优先采用成熟开源框架或事实标准协议（如 BPMN、OpenAPI、S3、SQL）。
2. 若云上已有托管服务，优先选择与开源协议兼容的托管实现，避免被厂商接口锁死。
3. 仅在业务有明显差异化时开发“薄适配层”，避免重写认证、工作流、对象存储、任务调度、报表引擎等通用基础能力。
4. 自研范围严格限制在业务规则本身，如库存分配策略、SKU 映射、审批规则组合、报税口径封装。

采用 Maven 多模块项目结构，按业务域拆分：

```

├── pom.xml                        # 父 POM，统一依赖版本
├── erp-common/                    # 公共模块
│   ├── erp-common-core/           # 核心工具、统一响应、异常定义
│   ├── erp-common-security/       # 安全框架封装
│   ├── erp-common-mybatis/        # MyBatis-Plus 配置
│   ├── erp-common-redis/          # Redis / Redisson 封装
│   ├── erp-common-storage/        # MinIO / OSS / S3 兼容封装
│   ├── erp-common-workflow/       # Flowable 统一封装
│   ├── erp-common-job/            # XXL-JOB 任务封装
│   ├── erp-common-report/         # POI / OpenPDF 导出封装
│   └── erp-common-integration/    # WebClient + Retry + 熔断封装
├── erp-system/                    # 系统管理 (用户、角色、权限、字典)
├── erp-product/                   # 产品管理 (产品、分类、BOM、包装、标签)
├── erp-material/                  # 原料管理 (原料、供应商)
├── erp-inventory/                 # 库存管理 (入库、出库、盘点、调拨)
├── erp-sales/                     # 销售管理 (客户、订单、退货)
├── erp-purchase/                  # 采购管理 (采购单、到货、应付)
├── erp-production/                # 生产管理 (工艺、工单、MRP)
├── erp-finance/                   # 财务管理 (价格、发票、报税)
├── erp-hr/                        # 人事管理 (员工、考勤、审批)
├── erp-integration/               # 外部集成 (电商、物流、发票平台)

```

**公共模块边界：**
- 仅封装第三方框架的接入规范、配置模板和通用适配器。
- 不在公共模块里重复实现上游框架已有能力，避免形成“二次框架”。

### 4.4 关键设计决策

#### 4.4.1 开源优先与非自研边界

- 审批采用 **Flowable**，不自研工作流引擎。
- 定时任务采用 **XXL-JOB**，不自建任务调度中心。
- 对象存储采用 **MinIO / OSS / S3 兼容接口**，不自建文件服务。
- 分布式锁采用 **Redisson**，不手写 Redis Lua 锁框架。
- 重试/熔断采用 **Spring Retry + Resilience4j**，不自写容错中间件。
- 报表导出采用 **Apache POI + OpenPDF**，不自研报表模板协议。
- Trace、指标、日志透传采用 **Micrometer / OpenTelemetry** 体系，不依赖手写 ThreadLocal 方案。

#### 4.4.2 单体 vs 微服务

**当前阶段采用模块化单体**，原因：
- 团队规模小，微服务运维成本高
- ERP 业务模块间耦合度高（如订单涉及库存、财务、物流）
- 单体内模块间调用无网络开销，事务管理简单

**未来拆分策略**：
- 当单模块代码量 > 5万行或团队 > 10人时考虑拆分
- 优先拆分"集成模块"（电商对接天然独立）
- 通过 RabbitMQ 事件驱动解耦做前期准备

#### 4.4.3 API 设计规范

```
基础路径: /api/v1/{module}/{resource}
示例:
  GET    /api/v1/product/products          # 产品列表（分页）
  GET    /api/v1/product/products/{id}     # 产品详情
  POST   /api/v1/product/products          # 创建产品
  PUT    /api/v1/product/products/{id}     # 更新产品
  DELETE /api/v1/product/products/{id}     # 删除产品
  POST   /api/v1/product/products/export   # 导出
```

**统一响应格式：**
```json
{
  "code": 200,
  "message": "success",
  "data": {},
  "timestamp": 1716220800000
}
```

**错误码体系：**
| 范围 | 模块 |
|------|------|
| 10000-19999 | 系统级错误 |
| 20000-29999 | 产品模块 |
| 30000-39999 | 库存模块 |
| 40000-49999 | 销售模块 |
| 50000-59999 | 财务模块 |
| 60000-69999 | 集成模块 |

#### 4.4.4 认证与授权

```
认证流程:
用户登录 → 验证凭据 → 签发 JWT (Access Token + Refresh Token)
                                    │
                        Access Token: 有效期 2h
                        Refresh Token: 有效期 7d, 存 Redis

API 请求:
Request → Nginx → Spring Security Filter → JWT 验证
    → 解析用户信息 → 权限校验 (RBAC) → Controller
```

**权限模型**：
- 基于 RBAC（用户-角色-权限）
- 支持数据权限（部门级、个人级）
- 支持字段级权限（敏感字段如成本价仅管理员可见）
- 操作审计日志（who/when/what/where）

**开源优先说明：**
- 初期直接基于 Spring Security 6 的标准过滤链、认证管理器、方法级鉴权能力实现。
- 不自写 OAuth2 / SSO 协议栈；若后续出现统一身份平台需求，优先接入 **Keycloak** 或 **Spring Authorization Server**。
- 数据权限只扩展业务查询条件，不复制一套新的鉴权框架。

#### 4.4.5 缓存策略

| 数据类型 | 缓存方式 | TTL | 更新策略 |
|----------|---------|-----|---------|
| 数据字典 | Redis String | 24h | 修改时主动失效 |
| 用户权限 | Redis Hash | 2h | 角色变更时刷新 |
| 产品信息 | Redis String | 1h | Cache Aside |
| 库存数量 | Redis + DB + Redisson 锁 | 实时 | Cache Aside + 显式锁保护 |
| 会话Token | Redis String | 2h/7d | 登出时删除 |

#### 4.4.6 消息队列使用场景

| 场景 | Exchange | 说明 |
|------|----------|------|
| 电商订单同步 | order.sync | 平台推送 → 消费入库 |
| 库存变动通知 | inventory.change | 库存变化 → 电商同步/预警 |
| 发票开具 | invoice.issue | 异步开票 |
| 操作日志 | audit.log | 异步写入日志表 |
| 报表生成 | report.generate | 耗时报表异步生成 |

#### 4.4.7 外部集成与容错策略

| 能力 | 方案 | 说明 |
|------|------|------|
| HTTP 客户端 | Spring WebClient | 统一请求、超时、日志、指标 |
| 重试 | Spring Retry | 针对限流、超时、瞬时失败做策略化重试 |
| 熔断/限流/隔离 | Resilience4j | 防止第三方故障拖垮主业务链路 |
| SDK 选择 | 优先官方 SDK，其次 REST 适配层 | 避免维护不必要的协议细节 |
| 适配层设计 | `adapter + mapper + callback handler` | 保留平台差异，但不污染核心业务 |

#### 4.4.8 报表与导出策略

- 报表展示层采用 **ECharts**，不自研图表引擎。
- Excel 导出采用 **Apache POI**，PDF 导出采用 **OpenPDF**。
- 首版不建设复杂报表设计器或 DSL，重点通过 SQL 聚合、物化视图和前端筛选满足经营分析需求。
- 如后续出现固定格式套打报表需求，优先评估 **JasperReports**，而不是自建模板编排语言。

**Tauri Rust 端职责边界（严格最小化）：**

- 不为通用能力编写大规模 Rust 业务逻辑

## 六、数据库架构设计

### 6.1 PostgreSQL 特性利用

| 特性 | 使用场景 |
|------|---------|
| **JSONB** | 产品动态规格、标签模板配置、电商平台原始数据 |
| **窗口函数** | 库存周转率、销售排行、财务汇总报表 |
| **CTE (WITH)** | BOM 递归展开、组织架构树查询 |
| **全文检索 (tsvector)** | 产品搜索、客户搜索（配合 zhparser 中文分词） |
| **分区表** | 库存流水表按月分区、操作日志表按月分区 |
| **行级锁 (FOR UPDATE)** | 库存扣减原子性保证 |
| **数组类型** | 权限标识符存储、标签列表 |
| **生成列** | 自动计算字段（如含税金额 = 不含税 × 税率） |

### 6.2 数据库规范

- 表名：小写下划线，如 `sale_order`
- 字段名：小写下划线，如 `created_at`
- 主键：UUID（`gen_random_uuid()`），避免自增 ID 暴露业务量
- 时间字段：统一使用 `TIMESTAMPTZ`（带时区）
- 软删除：`deleted` BOOLEAN + `deleted_at` TIMESTAMPTZ
- 审计字段：每表必含 `created_by`, `created_at`, `updated_by`, `updated_at`
- 索引命名：`idx_{表名}_{字段名}`
- 外键命名：`fk_{表名}_{关联表名}`

### 6.3 读写分离与扩展

- **初期**：单节点 PostgreSQL + 定期备份
- **中期**：主从复制，读操作分流到从库
- **远期**：按业务域分库（如财务独立库）

**配套原则：**
- 优先用 PostgreSQL 原生能力解决事务、一致性、报表聚合和全文检索问题。
- 在 PostgreSQL 能满足需求前，不额外引入 ES、时序库或专用搜索中间件，控制系统复杂度。

## 七、安全架构

### 7.1 网络安全

- 全站 HTTPS（TLS 1.3），Nginx 终止 SSL
- 内网服务间通信可用 HTTP（Docker 网络隔离）
- API 限流：Nginx `limit_req_zone` + 应用层令牌桶

### 7.2 应用安全

| 威胁 | 防护措施 |
|------|---------|
| SQL 注入 | MyBatis-Plus 参数绑定，禁止字符串拼接 SQL |
| XSS | 前端输入消毒，后端输出编码，CSP 头 |
| CSRF | JWT + SameSite Cookie + 自定义Header校验 |
| 越权 | 数据权限过滤（部门/个人），接口粒度鉴权 |
| 密码泄露 | BCrypt 加密存储，密码强度策略 |
| Token 泄露 | 短有效期 + Refresh Token 轮转 + 异常登录检测 |

### 7.3 数据安全

- 敏感字段加密存储（AES-256）：电商平台 Token、银行账号
- 数据库连接池加密
- 定期自动备份（pg_dump → MinIO/OSS/S3 兼容存储），保留 30 天
- 操作审计日志不可篡改（追加写入）

### 7.4 合规要求

- 符合《个人信息保护法》：员工信息、客户信息最小化收集
- 数据导出需权限控制与日志记录
- 支持数据删除请求（GDPR right to erasure）

## 十二、技术选型对比总结

### 12.1 端到端技术路线对比

| 维度 | 本方案 (Spring+React+Tauri) | 全 Rust 方案 | 全 JavaScript 方案 (Node+Electron) |
|------|----------------------------|-------------|-----------------------------------|

| 开发效率 | 极高（成熟生态+AI友好） | 中等 | 高 |
| 桌面端体积 | 小 (<10MB) | 小 (<10MB) | 大 (>50MB) |
| 多端代码复用 | 高（React Monorepo） | 低 | 高 |
| 国内招聘难度 | 低 | 高 | 低 |
| 运维复杂度 | 中 | 高 | 中 |
| 长期可维护性 | 高 | 中 | 中 |

### 12.2 核心能力开源组件清单

| 能力 | 首选方案 | 选择理由 |
|------|---------|---------|
| 审批引擎 | Flowable | BPMN 标准、生态成熟、支持可视化流程建模 |
| 定时任务 | XXL-JOB | 国内实践成熟、控制台完善、适合中小团队 |

| API 文档/SDK | Springdoc + openapi-generator | 前后端契约一致，减少重复维护 |
| Excel/PDF 导出 | Apache POI + OpenPDF | 社区成熟、格式支持稳定 |
| 桌面原生能力 | Tauri 官方插件生态 | 维护成本低，兼容性与升级路径清晰 |
| 移动原生能力 | Expo Modules | 避免大量自定义原生桥接开发 |

### 12.3 自研边界总结

**明确自研：**
- 库存锁定与分配规则
