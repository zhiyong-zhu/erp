## 一、项目概述

### 1.1 项目背景

随着企业业务向多终端、多渠道发展，传统 ERP 系统已难以满足"客户端+浏览器端+移动App端"三端协同管理的需求。当前企业面临的核心痛点：

| 痛点 | 现状 | 目标 |
|------|------|------|
| 数据割裂 | 各端数据独立，依赖人工同步 | 统一数据源，实时同步 |
| 电商脱节 | 平台订单与库存需人工核对 | 自动同步订单、库存、物流 |
| 标签打印低效 | 手工操作，易出错 | 一键打印，模板化管理 |
| 报税繁琐 | 依赖外部会计系统，数据不闭环 | 进销项自动汇总，一键申报 |
| 多端不统一 | 桌面/Web/手机体验割裂 | 统一交互体验，数据实时同步 |

### 1.2 项目目标

构建一套全渠道ERP系统，覆盖"产品-原料-工艺-采购-库存-价格-销售-报税-发票-员工"全链路，支持桌面端、Web端、移动端三端协同。

**核心目标：**
1. 实现核心业务全流程数字化覆盖
2. 集成主流电商平台，自动同步订单与库存
3. 支持包装规格管理与一键标签打印
4. 实现报税自动化，关联采购与销售数据形成闭环
5. 三端统一体验，数据实时同步

### 1.3 项目范围

**包含：**
- 产品管理、原料管理、工艺管理
- 采购管理、库存管理、销售管理
- 价格管理、财务管理（发票+报税）
- 员工管理、权限管理
- 电商平台集成（淘宝/天猫、京东、拼多多、抖音）
- 标签打印系统
- 报表与数据分析

**不包含（本期）：**
- 完整的 HR 系统（如薪酬核算、社保公积金）
- CRM 深度功能（如营销自动化）
- 独立的 WMS 系统（仅基本库位管理）
- 跨境电商（海关报关、跨境物流）

### 1.4 首版上线范围（V1.0）

为控制首版交付风险，V1.0 以“打通主数据、订单、库存、发票、报税、电商同步”闭环为目标，默认交付全部 P0 功能和业务闭环所必需的 P1 功能。

| 业务域 | 首版上线内容 | 暂缓到后续版本 |
|------|-------------|---------------|
| 产品 | 分类、产品/SKU、包装、标签打印、导入导出 | 高级标签设计器、复杂替代料策略 |
| 原料/采购 | 原料、供应商、采购单、到货验收、采购入库、采购退货、应付统计 | 供应商自动评分、智能比价推荐 |
| 库存 | 多仓库、多货位、出入库、盘点、调拨、库存预警 | 高级波次路径优化、自动补货算法 |
| 生产 | 工艺路线、生产工单、报工、完工入库 | 甘特图排程、MES 深度集成 |
| 销售/电商 | 客户、手工订单、电商订单同步、发货、退货、物流回传 | CRM 深度能力、售后客服工作台 |
| 财务 | 发票台账、税额汇总、申报表导出、税务提醒、应收应付统计视图 | 总账、自动凭证、固定资产、银企直连 |
| 报表/系统 | 仪表盘、核心经营报表、RBAC、审批、通知、编号规则 | 自助 BI、复杂报表建模 |

### 1.5 业务成功指标

| 指标 | 上线后 3 个月目标 | 说明 |
|------|------------------|------|
| 电商订单自动入单率 | ≥ 95% | 正常订单无需人工补录 |
| 库存账实准确率 | ≥ 98.5% | 以盘点结果为准 |
| 标签打印一次成功率 | ≥ 99.5% | 包含桌面端本地打印与 Web 云打印 |
| 采购到货入库及时率 | ≥ 95% | 到货当日完成验收并入库 |
| 报税准备耗时下降 | ≥ 70% | 相比手工整理发票与税额汇总 |
| 关键业务线上化覆盖率 | ≥ 90% | P0 流程不再依赖 Excel/手工台账 |

## 三、功能模块详细需求

### 3.0 核心业务闭环

#### 3.0.1 销售闭环

产品建档 → 定价生效 → 创建销售订单/同步电商订单 → 锁定库存 → 出库发货 → 物流回传 → 开具销项发票 → 汇总税额 → 销售/财务报表。

#### 3.0.2 采购与生产闭环

安全库存预警/MRP 建议 → 创建采购单 → 审批 → 到货验收 → 原料入库 → 领料生产 → 报工完工 → 成品入库 → 供销售发货。

#### 3.0.3 异常闭环

缺货、退货、盘盈盘亏、价格低于阈值、发票差异、电商同步失败等异常必须形成“告警/审批/处理/留痕”闭环，不能只停留在消息提醒层面。

### 3.1 产品管理

#### 3.1.1 功能概述
管理企业所有产品信息，包括分类、规格、BOM、包装、标签。

#### 3.1.2 功能清单

| 功能 | 优先级 | 说明 |
|------|--------|------|
| 产品分类管理 | P0 | 多级分类树，支持无限层级 |
| 产品信息维护 | P0 | 基础信息、多规格SKU、图片 |
| BOM 物料清单 | P1 | 多级 BOM，支持替代料 |
| 包装规格管理 | P0 | 单品→内盒→外箱层级定义 |
| 标签模板设计 | P1 | 可视化设计器，支持条码/二维码 |
| 标签打印 | P0 | 一键打印（桌面端本地/Web端云打印） |
| 产品导入导出 | P1 | Excel 批量导入导出 |

#### 3.1.3 用户故事

**US-P01：产品创建**
> 作为产品专员，我需要创建新产品并设置多规格SKU，以便系统能管理不同规格的库存和价格。

验收标准：
- 可设置产品基础信息（名称、编码、分类、单位、图片）
- 可添加多个规格属性（如颜色、尺寸），自动生成SKU组合
- 创建后产品状态为"草稿"，审核通过后变为"在售"
- 编码自动生成，规则可配置（如前缀+分类码+流水号）

**US-P02：包装规格定义**
> 作为仓管主管，我需要为产品定义包装层级（单品→内盒→外箱），以便出库时自动计算装箱数量。

验收标准：
- 每个产品可定义 1-3 层包装（单品、内盒、外箱）
- 每层需填写：名称、装入数量、重量、尺寸、条码
- 支持自动计算：1外箱 = N内盒 = M单品
- 包装信息可关联到标签模板

**US-P03：标签打印**
> 作为仓管员，我需要选择产品和包装规格后一键打印标签，以便贴在包装上。

验收标准：
- 选择产品 → 选择包装层级 → 选择标签模板 → 输入打印数量
- 桌面端：直接调用本地打印机打印，无需浏览器打印对话框
- Web端：生成 PDF 预览后打印，或通过云打印服务
- 支持打印预览
- 支持批量打印（多产品多规格一次性打印）

#### 3.1.4 业务规则

| 规则编号 | 规则描述 |
|----------|---------|
| BR-P01 | 产品编码全局唯一，创建后不可修改 |
| BR-P02 | 产品存在未完成订单或库存时不可删除，仅可停用 |
| BR-P03 | BOM 层级不超过 10 层，防止循环引用 |
| BR-P04 | SKU 变体生成后，已有库存的规格不可删除 |

#### 3.1.5 产品状态机

```
草稿 → [提交审核] → 审核中 → [审核通过] → 在售
                          ↓                    ↓
                    [审核驳回]           [停用]
                          ↓                    ↓
                        草稿               已停用
                                              ↓
                                         [启用]
                                              ↓
                                            在售
```

### 3.2 原料管理

#### 3.2.1 功能清单

| 功能 | 优先级 | 说明 |
|------|--------|------|
| 原料分类 | P0 | 多级分类 |
| 原料信息维护 | P0 | 基础信息、规格、默认供应商 |
| 供应商管理 | P0 | 供应商档案、联系人、资质 |
| 供应商报价 | P1 | 记录历史报价，比价参考 |
| 安全库存预警 | P0 | 低于阈值自动提醒 |
| 批次追溯 | P1 | 原料批次与成品批次关联 |

#### 3.2.2 用户故事

**US-M01：安全库存预警**
> 作为采购经理，当原料库存低于安全库存时，我需要收到预警通知，以便及时采购补货。

验收标准：
- 每种原料可设置安全库存阈值
- 库存低于阈值时，自动生成预警消息推送给采购经理
- 预警信息包含：原料名称、当前库存、安全库存、推荐采购量
- 移动端同步推送通知

#### 3.2.3 业务规则

| 规则编号 | 规则描述 |
|----------|---------|
| BR-M01 | 原料编码全局唯一，停用后不可被复用 |
| BR-M02 | 存在库存、未完成采购单或被 BOM 引用的原料不可删除，仅可停用 |
| BR-M03 | 启用安全库存预警前，必须维护安全库存、采购周期和默认供应商 |
| BR-M04 | 原料批次需可追溯到采购批次、入库时间、供应商及关联成品批次 |

### 3.3 采购管理

#### 3.3.1 功能清单

| 功能 | 优先级 | 说明 |
|------|--------|------|
| 采购申请 | P0 | 手动创建或 MRP 自动生成 |
| 采购订单 | P0 | 审批后生成正式采购单 |
| 到货验收 | P0 | 扫码验收，数量/质量检验 |
| 采购入库 | P0 | 验收通过后自动入库 |
| 采购退货 | P1 | 质量不合格退货流程 |
| 应付管理 | P1 | 应付账款、付款计划 |
| 采购报表 | P1 | 采购统计、供应商考核 |

#### 3.3.2 用户故事

**US-PU01：采购全流程**
> 作为采购专员，我需要从创建采购单到验收入库的全流程管理，确保原料及时补充且账目清晰。

验收标准：
- 采购单可手动创建或从 MRP 建议转换
- 采购单需经采购经理审批
- 到货后仓管扫码验收，录入实收数量
- 验收通过自动生成入库单
- 入库后自动生成应付凭证
- 采购单关联进项发票，为报税提供数据

#### 3.3.3 采购单状态机

```
草稿 → [提交] → 待审批 → [审批通过] → 已审批 → [发送供应商] → 已下单
                    ↓                                              ↓
              [审批驳回]                                     [部分到货]
                    ↓                                              ↓
                  草稿                                        部分到货
                                                                   ↓
                                                              [全部到货]
                                                                   ↓
                                                              已完成
```

#### 3.3.4 业务规则

| 规则编号 | 规则描述 |
|----------|---------|
| BR-PU01 | 采购单审批通过前不得生成正式到货验收单 |
| BR-PU02 | 到货验收数量默认不得超过采购数量，超量收货需走异常审批 |
| BR-PU03 | 不合格到货需进入待处理状态，支持退货、让步接收或补货 |
| BR-PU04 | 被停用或资质过期的供应商不可新建采购单 |
| BR-PU05 | 采购单、验收单、退货单全程保留审批人与时间戳，供审计追溯 |

#### 3.3.5 采购异常处理

| 场景 | 处理要求 |
|------|---------|
| 部分到货 | 支持多次收货，订单保持“部分到货”直到全部收齐 |
| 超量到货 | 默认拦截，允许有权限人员审批放行 |
| 质量不合格 | 记录不合格原因、数量、处理意见，并可生成采购退货 |
| 超期未到货 | 自动提醒采购专员与采购经理 |
| 发票金额与采购金额差异 | 差异超过阈值时需财务人工确认 |

### 3.4 库存管理

#### 3.4.1 功能清单

| 功能 | 优先级 | 说明 |
|------|--------|------|
| 多仓库管理 | P0 | 支持多仓库、多货位 |
| 入库管理 | P0 | 采购入库、生产入库、退货入库 |
| 出库管理 | P0 | 销售出库、领料出库、调拨出库 |
| 库存盘点 | P0 | 支持全盘、抽盘、动态盘点 |
| 库存调拨 | P1 | 仓库间调拨，需审批 |
| 序列号管理 | P2 | 高值品逐件追踪 |
| 波次拣货 | P1 | 多订单合并拣货，路径优化 |
| 库存预警 | P0 | 安全库存、呆滞库存预警 |

#### 3.4.2 用户故事

**US-I01：扫码出库**
> 作为仓管员，我需要根据出库单扫码校验并确认出库，确保发出的商品与订单一致。

验收标准：
- 移动端或桌面端扫码，自动匹配出库单明细
- 扫码后实时显示：已拣数量/应拣数量
- 数量不匹配时报警提示
- 全部拣完后确认出库，库存实时扣减
- 出库完成自动触发物流单号回传（电商订单场景）

**US-I02：库存盘点**
> 作为仓库主管，我需要创建盘点计划并由仓管员执行，自动生成盈亏报告。

验收标准：
- 可按仓库/货位/分类创建盘点计划
- 仓管员移动端扫码录入实盘数量
- 系统自动计算盈亏差异
- 盈亏超过阈值需主管审批
- 审批通过后自动调整账面库存

#### 3.4.3 业务规则

| 规则编号 | 规则描述 |
|----------|---------|
| BR-I01 | 出库时库存不足则拒绝操作并提示 |
| BR-I02 | 库存扣减采用悲观锁（SELECT FOR UPDATE），防止超卖 |
| BR-I03 | 电商订单自动锁定库存（待发货状态），不可被其他订单占用 |
| BR-I04 | 盘点期间该仓库/货位暂停出入库操作 |
| BR-I05 | 库存流水不可删除、不可修改，仅追加 |

#### 3.4.4 盘点单状态机

```
草稿 → [开始盘点] → 盘点中 → [提交盘点结果] → 待审批 → [审批通过] → 已完成
                         ↑                        │
                         └── [退回修改] ←─────────┘
                                                  │
                                            [审批驳回]
                                                  │
                                                 草稿
```

#### 3.4.5 调拨单状态机

```
草稿 → [提交] → 待审批 → [审批通过] → 已审批 → [出库确认] → 调出完成 → [入库确认] → 已完成
                  │
            [审批驳回]
                  │
                 草稿
```

#### 3.4.6 高值品与波次拣货要求

- 序列号管理按 SKU 维度启用，仅对高值品、售后敏感品或监管要求商品开启。
- 启用序列号的 SKU 在入库、出库、退货时必须逐件采集。
- 波次拣货首版支持按仓库、平台、发货时间窗手动分波，不要求自动路径寻优。
- 波次拣货结果需支持打印拣货单、查看拣货进度与异常缺货回退。

#### 3.5.4 业务规则

### 3.8 电商平台集成

#### 3.8.1 集成范围

| 平台 | 优先级 | 集成内容 |
|------|--------|---------|
| 淘宝/天猫 | P0 | 订单同步、库存同步、物流回传、退款同步 |
| 京东 | P0 | 订单同步、库存同步、物流回传 |

| 抖音 | P1 | 订单同步、库存同步、物流回传 |

#### 3.8.2 集成功能详述

**订单同步：**
- 定时拉取 + 消息推送双模式
- 新订单、付款、退款、关闭状态同步
- 订单金额、优惠、运费明细拆解
- 异常订单（缺货、地址异常）自动标记

**库存同步：**
- 库存变动实时/准实时同步到各平台
- 支持安全库存比例（如仅同步可用库存的 90%）
- 多店铺库存分配策略（共享池 / 独立池）

**物流回传：**
- 发货后自动回传物流单号到平台
- 支持主流快递公司编码映射

#### 3.8.3 异常处理

| 异常场景 | 处理方式 |
|----------|---------|
| 平台 API 限流 | 指数退避重试，切换到备用 Token |
| 平台 Token 过期 | 自动刷新，失败则通知运营重新授权 |
| 订单同步失败 | 记录失败队列，支持手动重试 |
| 库存同步冲突 | 以本地系统为准，覆盖平台库存 |
| 平台 API 升级 | 适配层版本化，新旧版本并行过渡 |

#### 3.8.4 主数据映射与对账要求

- 平台商品与本地 SKU 必须建立映射关系后，方可执行库存同步。
- 未匹配 SKU 的订单需进入“待匹配”队列，不得直接入正式销售流程。
- 系统需提供订单对账、库存对账、退款对账三类每日对账清单。
- 运营人员可按店铺、时间范围发起手工补同步并查看重试结果。

#### 3.9.3 用户故事

**US-F01：报税自动计算**
> 作为财务经理，我需要系统自动汇总本月销项税与进项税，计算应缴税额，避免手工统计遗漏。

验收标准：
- 每月自动汇总：销项发票税额合计、进项发票税额合计
- 自动计算：应缴增值税 = 销项税额 - 进项税额 - 上期留抵

- 可导出标准申报表格式（Excel/PDF）

#### 3.9.4 业务规则

| 规则编号 | 规则描述 |
|----------|---------|
| BR-F01 | 进项发票需在 360 天内认证（勾选确认）方可抵扣 |
| BR-F02 | 红字发票需关联原蓝字发票 |
| BR-F03 | 发票金额与业务单据金额差异超过 0.01 元需人工确认 |

#### 3.9.5 发票状态机

```

                  │                     │
                  │               [冲红（开红字发票）]
                  │                     │
                  └────── 已冲红 ←──────┘
```

**状态值对照：**

| 状态 | 值 | 适用方向 | 含义 |
|------|-----|---------|------|
| 草稿 | 0 | 全部 | 已录入未开具/未收到 |
| 已开具 | 1 | 销项 | 已开出给客户 |

#### 3.9.6 报税汇总状态机

```
计算中 → [确认] → 已确认 → [申报] → 已申报
             │
       [驳回（重新计算）]
             │
           计算中
```

#### 3.9.7 财务范围边界

**本期包含：**
- 销项/进项发票管理
- 发票台账与税额汇总
- 税务申报表导出与提醒
- 收付款状态记录
- 应收应付统计视图与经营报表口径

**本期不包含：**
- 总账、自动凭证、结账结转
- 固定资产、费用报销、资金计划
- 银企直连、网银回单自动采集
- 薪资核算与人力成本归集

#### 3.9.8 关键财务闭环

采购入库/销售发货只是业务完成，不代表财务闭环完成；首版要求至少打通“业务单据 → 发票 → 税额汇总 → 申报表 → 报表展示”的闭环，并对收付款状态进行可视化跟踪。

## 五、数据迁移需求

### 5.1 迁移范围

如有旧系统，需迁移以下数据：
- 产品主数据（编码、名称、规格）
- 客户与供应商档案
- 历史库存数据（初始化库存）
- 历史订单数据（可选，用于报表）

### 5.2 迁移策略

| 阶段 | 内容 |
|------|------|
| 数据清洗 | 旧数据去重、格式标准化 |
| 映射设计 | 旧字段与新字段映射关系 |
| 试迁移 | 非生产环境验证 |
| 正式迁移 | 停机窗口内执行 |
| 数据校验 | 抽样核对，总量对比 |

### 5.3 迁移验收标准

- 主数据数量校验：产品、客户、供应商、仓库数量与旧系统一致或有签字说明
- 库存金额校验：按仓库、SKU 汇总后的数量与金额差异在允许阈值内
- 历史订单抽样校验：至少覆盖近 3 个月订单，字段准确率 ≥ 99%
- 财务基础校验：发票台账和税额汇总数据与旧系统/手工台账核对一致
- 迁移结果需由业务负责人、财务负责人、项目经理共同签字确认

### 5.4 回退策略

- 正式迁移前冻结旧系统新增关键主数据和业务单据录入窗口
- 迁移前生成数据库快照和导出包，确保可回退
- 上线后若出现 P0 阻断问题，可切回旧系统并补录切换窗口内数据
- 回退流程、责任人、决策口径需在正式迁移前书面确认

## 六、验收标准

### 6.1 功能验收

- 全部 P0 功能通过测试用例验证
- 核心业务流程端到端跑通
- 电商订单同步成功率 ≥ 99%
- 标签打印成功率 ≥ 99.5%

### 6.2 性能验收

- 500 并发下 API 平均响应时间 < 200ms
- 页面首屏加载 < 2s
- 无内存泄漏、无 CPU 持续高负载

### 6.3 安全验收

- 通过 OWASP Top 10 安全扫描
- 越权测试全部通过
- 敏感数据无明文暴露

### 6.4 核心业务场景验收

| 场景编号 | 场景描述 | 验收结果要求 |
|---------|---------|-------------|
| UAT-01 | 产品建档 → 包装定义 → 标签打印 | 主数据、模板、打印链路全打通 |
| UAT-02 | 原料预警 → 创建采购单 → 审批 → 验收入库 | 预警、审批、库存入账一致 |
| UAT-03 | 手工销售订单 → 锁库 → 发货 → 物流回传 | 订单状态、库存、物流状态一致 |
| UAT-04 | 电商平台订单同步 → 发货 → 平台回传 | 平台与本地订单状态一致 |
| UAT-05 | 销售退货 → 库存恢复 → 退款处理 | 逆向库存、退款、报表回冲正确 |
| UAT-06 | 发票开具/录入 → 税额汇总 → 申报表导出 | 发票、税额、报表口径一致 |
| UAT-07 | 盘点盈亏审批 → 库存调整 → 审计留痕 | 账实变更和日志可追溯 |

- 已完成主数据初始化和关键用户培训
- 已完成电商店铺授权、打印机联调、消息通道联调
- 已完成至少 1 轮全链路 UAT 和 1 轮试迁移
- P0/P1 阻断缺陷为 0，P2 缺陷有明确规避方案
- 上线值班表、回退预案、联系人清单已确认

## 七、约束与假设

### 7.1 约束条件

- 后端技术栈限定为 Spring Boot + PostgreSQL
- 前端技术栈限定为 React 生态
- 桌面端使用 Tauri（非 Electron）
- 初期团队规模 3-5 人
- 首版交付周期 4-5 个月

### 7.2 假设

- 电商平台 API 在开发期间保持稳定
- 企业已有阿里云/华为云等云服务账号
- 用户具备基本的电脑操作能力
- 标签打印机支持网络连接或 USB 直连
- 税务政策在开发期间无重大变化

### 7.3 外部依赖

| 依赖项 | 说明 |
|-------|------|
| 电商平台 API | 订单拉取、库存同步、物流回传依赖平台开放能力 |
| 发票/税务平台 | 数电票、发票查验、申报表口径依赖第三方或税局接口 |
| 物流平台 | 物流单号查询、快递公司编码映射、回传能力 |
| 打印机与驱动 | 桌面端本地打印效果依赖打印机品牌、驱动和指令兼容性 |
| 消息通道 | 邮件、企业微信/钉钉、移动推送服务可用性 |

### 7.4 待确认事项

- 首版是否需要覆盖京东、淘宝之外的平台全部正式上线
- 财务模块中收付款记录与应收应付统计的细化口径
- 价格审批阈值、盘点盈亏审批阈值、超量收货阈值的业务参数
- 打印机品牌范围、标签纸规格、是否需要统一采购硬件
